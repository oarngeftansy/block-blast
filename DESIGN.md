# Block Blast 原型 · 难度与关卡设计文档

本原型包含两种模式,共用同一执行底座(8×8 棋盘、形状库、整行/列消除、计分、道具),仅在**发牌来源**与**胜负 / 难度判定**两处分叉:

- **无尽模式**:闭环动态难度(DDA)——系统持续读玩家、追着调发牌难度,无限、几乎不"输"。
- **关卡模式**(Cube Busters 式):前馈式关卡编排——难度前置写死在关卡数据里,有限、可通关,靠步数制造失败态。

---

## 一、无尽模式 · 算法

把"难度"当受控量、把玩家表现当反馈信号的闭环控制系统。四层 + 一个全局增益。

### ① 玩家分层(ASS)
- 前 3 局分别取 普通 / 困难 / 极难 的得分,均值记为 `ass`。
- 分层:`ass < 1000` 低阶,`> 1500` 高阶,其间(1000–1500)中阶。
- 每个破纪录局组结束后复评。

### ② 调度
- `DIFF_CYCLE` 十格序列 `[普通,困难,极难,普通,困难,困难,普通,普通,极难,普通]` 按局推进名义难度。
- 走完一圈进入破纪录节点局组(`recordPhase` 1→3);破纪录或走满 3 局则复评、开新一轮。
- 以(层级 × 名义难度)查表选 L 模板(L1…L5),缺项按 普通→困难→极难→破纪录 兜底。

### ③ 执行
- L 模板展开成长度 16 的 **DG 序列**(局内难度曲线),每档停留 2 轮发牌。
- 每档 DG 映射一个**目标落点区间** `[lo, hi]`:发出的方块在当前局面上的合法落点数应落入其中,越少越逼死。

  | DG | DG1 | DG2 | DG3 | DG4 | DG5 | DG6 | DGS |
  |----|-----|-----|-----|-----|-----|-----|-----|
  | 落点区间 | 22–52 | 15–32 | 10–21 | 6–14 | 3–9 | 1–5 | 0–3 |

- 发牌:按权重采样 40 个候选形状,以「落点贴合区间 + 近期去重惩罚」为代价函数择优,取前 6 名随机一个;一轮发 3 块且互不重复(近期历史长度 8)。

### ④ 微观跳转(实时反馈)
| 触发条件 | 动作 |
|---|---|
| **空放**(连续落子未消)达阈值(普 4 / 难 5 / 极 6) | 降 1 档(救济) |
| 连击达阈值(普 25 / 难 20 / 极 15) | 升 1 档(加压) |
| 全清 | 升 2 档 × 2 轮 |
| 破纪录组 · 棋盘占用率 ≥ 85% | 降难助攻 |
| 破纪录组 · 本局分 ≥ best × 1.2 | 切 DG6/DGS 收尾增难 |

偏移叠加到原定 DG → 实际发牌档。

### ⑤ 全局增益
- `MATCH_DG_BOOST = +1`:仅无尽发牌时把整条 DG 曲线平移一档(用于整体提难;想更难调到 +2)。

### 计分
落子每格 +1;消除每格 ×3 × 行数倍率 × 连击乘法(1 + 连击 × 0.5);连续 3 步未消则连击归零。

> **例外**:第 1 局走新手发牌(专发 2×2 / 3×3 等方正大块并贴合当前空位智能匹配),不接入此闭环——给新玩家"极致爽感"的第一关。

> **本质**:玩家表现是自变量、发牌难度是因变量,系统持续追踪玩家,把其钉在心流带内,无终止条件。

## 二、关卡模式 · 算法

难度前置写死在关卡数据里,不做玩家自适应。

- **关卡 = 预置对称布局**:每关一组 8×8 字符,`.`空 `#`普通块 `G`宝石 `S`星 `L`叶 `O`齿轮(障碍) `C`糖条 → 解析成起手棋盘 + 元素表。
- **目标 + 步数**:各元素配额 + 步数预算。集齐即胜;步数耗尽即负。
- **消除即收集**:拖放填满整行/列触发消除,被消格上的元素计入对应配额;障碍同理(消其行/列才解除)。
- **供应维持**:每次消除后,单格收集物与糖条按**起手密度补种**(故配额可高于一眼可见量);齿轮为消耗型、不补。
- **发牌**:固定 DG1(宽松好放),不接入 DDA、不吃 `MATCH_DG_BOOST`。
- **脱困**:三块都放不下时,自动刷新成"在当前局面至少有一个合法落点"的方块(免费、不卡死);仅当棋盘饱和(任何形状都放不下)才回退到道具脱困 / 判负。
- **道具**:💣 炸 3×3;✚ 清整行 + 整列(即爆破 booster);清除同样收集元素。
- **评星**:剩余步数占比 ≥ 0.4 三星、≥ 0.15 两星、否则一星;本地持久化。

> **本质**:开环、难度预定、有限且可通关;体验曲线靠人工编排(锯齿状张弛),变现挂在失败态(步数耗尽)上。

## 三、关卡模式 · 每关设计思路

**通用准则**:一关引入一种新元素再叠加;对称保证可读、公平、补种后分布均衡;元素摆位由"该靠哪条线清掉"反推——收集物放易清处、障碍放最难填的角 / 边、糖条横铺整行做高回报;起手低填充保证落子空间;配额↑ + 障碍↑ + 相对步数↓ 拉难度曲线。

### c1 采集 — 宝石 8 / 22 步
```
........
........
..G..G..
.G.GG.G.
.G.GG.G.
..G..G..
........
........
```
纯教学关,只有宝石。集中在中部 4 行、上下各留两整行空 → 随手补满中部行即消除收集,门槛最低,教会"消整行/列 = 收集"的核心循环。配额(8)低于摆放量(12)、步数极松,几乎不可能输,专门用来上手。

### c2 齿轮 — 宝石 8 + 齿轮 4 / 26 步
```
O......O
........
..G..G..
.G....G.
.G....G.
..G..G..
........
O......O
```
引入障碍。**齿轮放四角是核心**:角格同时属于最外行与最外列,而边行/列最长、最难填满,所以齿轮天然是"难清的硬目标"。宝石在内圈相对好收。玩家被迫额外去攻边行/列,制造与收宝石不同的第二种节奏。四角对称 = 两侧难度均衡。

### c3 糖条 — 宝石 4 + 糖条 10 / 26 步
```
........
.CCCCCC.
........
...GG...
...GG...
........
.CCCCCC.
........
```
引入多格条状收集物。糖条横铺第 1/6 行、**只缺两端两格** → "补两格清一整条糖"的高回报点,教玩家"凑满一行 = 一次性大量收集"。宝石压在正中 2×2 作次要目标。糖条设为主配额(10)、宝石点缀(4)。

### c4 混合 — 星 6 + 齿轮 4 + 糖条 4 / 30 步
```
O......O
.S.CC.S.
.S....S.
........
........
.S....S.
.S.CC.S.
O......O
```
首次三元素组合(用星替宝石避免重复)。**星沿左右第 1/6 列排布 → 诱导玩家清"列";糖条在上下行;齿轮守四角**——三种目标分别奖励列清、行清、边清三种动作。中间两整行刻意全空作落子缓冲,压低组合关的堵死风险。

### c5 硬核 — 宝石 6 + 星 4 + 齿轮 4 + 糖条 6 / 36 步
```
O.G..G.O
.CC..CC.
G..SS..G
........
........
G..SS..G
.CC..CC.
O.G..G.O
```
综合考核。四类目标分布在**角(齿轮)、边中点(宝石)、内核(星)、整行段(糖条)**四种位置 → 逼玩家规划"先清哪条线、各条线服务哪个目标"。步数 36 看似宽,但合计配额 20 且齿轮难清,实际偏紧。中间两行留空保证可玩性。

## 四、两模式对照

| 维度 | 无尽模式 | 关卡模式 |
|---|---|---|
| 控制论 | 闭环 / 反馈 | 开环 / 前馈 |
| 难度来源 | 系统实时追踪玩家 | 作者预先写死 |
| 谁适应谁 | 系统适应玩家 | 玩家逼近设计 |
| 时长 | 无限 | 有限、可通关 |
| 失败态 | 几乎无(走死才停) | 步数耗尽(= 付费触点) |
| 核心驱动 | 心流 / 留存 | 进度 / 收集 / 成就 |
| 发牌 | DDA 调度 + 全局增益 | 固定 DG1 + 脱困刷新 |
| 难度旋钮 | ASS 层级 / DG 曲线 / 微观跳转 / BOOST | 布局密度 / 配额 / 步数预算 / 元素组合 |
| 变现 | 复活、皮肤、去广告 | 步数、道具、命、关卡包、通行证 |

## 五、关键可调参数速查

| 参数 | 位置 | 含义 |
|---|---|---|
| `TIER_THRESHOLDS` | 分层 | 低/高阶分数线(1000 / 1500) |
| `DIFF_CYCLE` | 调度 | 名义难度循环序列 |
| `TEMPLATES` / `TEMPLATE_TABLE` | 模板 | L 模板的 DG 序列与查表 |
| `DG_PLACEMENT` | 执行 | 各 DG 的目标落点区间 |
| `DG_ROUNDS` | 执行 | 每档 DG 停留轮数(2) |
| `MICRO_TRIGGERS` | 反馈 | 各难度的空放 / 连击阈值 |
| `MATCH_DG_BOOST` | 增益 | 无尽整体提难档数(+1) |
| `PUZZLES[].board` | 关卡 | 每关对称布局 |
| `PUZZLES[].goals` | 关卡 | 各元素配额 |
| `PUZZLES[].moves` | 关卡 | 步数预算 |
